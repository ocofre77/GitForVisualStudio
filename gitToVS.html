Utilice repositorios Git para alojar y colaborar en su código utilizando Visual Studio Team Services (VSTS) o Team Foundation Server (TFS).

Clona un repositorio existente de Git

Cree una copia local completa de un repositorio existente de Git clonándolo. Al clonar un repositorio, se descargan todas las confirmaciones y ramas en el repositorio y se establece una relación con el repositorio existente que se clonó. Use esta relación para interactuar con el repositorio existente, presionando y tirando de los cambios para compartir el código con su equipo.

Guarde Trabajo con Commits

Git no realiza instantáneamente un snapshot de tu código cuando realizas cambios en los archivos de tu repositorio. Debe informar a Git exactamente qué cambios desea agregar a la siguiente instantánea organizando esos cambios. Después de organizar sus cambios, cree un compromiso para guardar la instantánea en su repositorio.

El flujo de trabajo de Git utiliza ramas para aislar el trabajo hasta que esté listo para fusionar los cambios en su rama predeterminada, como master. La solicitud de extracción es el proceso colaborativo que le permite al resto del equipo hablar sobre los cambios en una sucursal y acordar fusionarlos una vez que todos lo aprueben. Utilice las solicitudes de extracción para obtener retroalimentación temprana de los demás sobre el trabajo en progreso, incluso si no está listo para fusionar los cambios en otra rama.

En esta imagen, la rama púrpura se fusiona en la rama azul a través de una solicitud de extracción. Los revisores debaten los cambios en la solicitud de extracción antes de fusionar el código. Cuando completa la solicitud de extracción, hay una confirmación de fusión (aquí se ve como la confirmación azul completada) donde los cambios en la rama púrpura ahora se fusionan en la rama azul.

Historial de revisión

Git usa la información de referencia principal almacenada en cada commit para administrar un historial completo de su desarrollo. Revise este historial de confirmaciones para saber cuándo se realizaron los cambios en los archivos y determinar las diferencias entre las versiones de su código.

El uso de Git de ramas de características y fusiones a través de solicitudes de extracción significa que el historial de confirmación de su desarrollo no forma una línea recta y cronológica. Cuando utiliza el historial para comparar versiones, piense en términos de cambios de archivos entre dos confirmaciones en lugar de cambios de archivos entre dos puntos en el tiempo. Un cambio reciente en un archivo en la masterrama puede haber venido de una confirmación creada hace dos semanas en una rama de características, pero solo se fusionó ayer.
